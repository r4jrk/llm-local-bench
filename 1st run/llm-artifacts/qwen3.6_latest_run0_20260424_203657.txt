For a Spring Boot SaaS integrating with Poland’s KSeF e-invoicing API, I recommend a decoupled, distributed architecture using Redis-backed rate limiting and Kafka for request queuing.

**IP+NIP Rate Limiting**
KSeF enforces per-taxpayer (NIP) and per-IP quotas. We treat NIP as the primary limiter and IP as a secondary fallback. Using Redisson’s distributed token buckets with Lua scripts, we atomically check both keys to avoid race conditions:
```java
boolean nipOk = nipLimiter.tryAcquire(1);
boolean ipOk = ipLimiter.tryAcquire(1);
if (!nipOk || !ipOk) { rejectOrQueue(); }
```
NIP buckets align to KSeF’s stated quotas (e.g., 500 req/min), while IP buckets use a lower shared threshold to throttle cross-tenant abuse. `X-Forwarded-For` headers resolve NAT/proxy IP blurring.

**Queue Architecture**
Inbound POSTs publish to a Kafka topic (`ksef-submissions`) partitioned by `NIP`. This guarantees tenant isolation and ordered processing. A scalable consumer group fetches messages, applies rate limits, and forwards payloads to KSeF. Success writes to a results cache; failures route to a DLQ. Kafka’s durability enables replay during outages, while consumer concurrency scales linearly with partition count.

**Retry & Backoff Strategy**
On HTTP 429 or transient network errors, we apply jittered exponential backoff: `delay = min(base * 2^attempt + random(0, 500ms), maxBackoff)`. Resilience4j wraps the KSeF client, capping at 5 attempts. Each retry records attempt metadata for tracing. Exhausted retries enter the DLQ.

```text
FUNCTION submitToKsef(req):
    IF NOT rateLimiter.tryAcquire(req.nip, req.ip):
        publishToQueue(req); return
    TRY:
        resp = httpClient.post("/ksef/v1/submit", req)
        IF resp.status == 429:
            retry(req) WITH backoff
        CATCH Timeout/5xx:
            circuitBreaker.recordFailure()
            publishToQueue(req) // async retry
    END TRY
END FUNCTION
```

**Tradeoffs**
- *Distributed vs Local:* Redis ensures cross-instance consistency but adds ~2ms latency and requires high-availability setup. Local caches (Caffeine) are faster but risk over-commit under scale.
- *Queue Decoupling:* Kafka absorbs spikes and guarantees delivery, but introduces 10–50ms latency. Real-time workflows may need a synchronous fallback path for critical invoices.
- *NIP vs IP Scoping:* Primary NIP limiting respects tenant quotas accurately, but shared IP limits can penalize tenants behind cloud NAT. Mitigated via dynamic IP bucket sizing and proxy-aware header parsing.
- *Operational Cost:* Kafka + Redisson increases infrastructure complexity and SRE burden. For moderate scale, Spring’s built-in `RateLimiter` + RabbitMQ suffices, but Kafka provides superior partitioning, audit trails, and replay capabilities required for compliant e-invoicing systems.

This design enforces quota compliance, isolates tenants, and survives KSeF instability while remaining aligned with Spring Boot’s reactive and cloud-native patterns.
