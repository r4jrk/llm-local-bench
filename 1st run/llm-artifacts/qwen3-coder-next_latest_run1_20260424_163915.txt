**Rate Limiting System for Spring Boot SaaS (KSeF Integration)**

We implement a two-tier rate limiting strategy: *client-level* (IP+NIP) and *system-level* (per-tenant & global), using Redis for distributed counting.

**Core Components:**
1. **Rate Limit Filter** (`RateLimitFilter`): Intercepts requests, extracts `X-Forwarded-For` (IP) and `X-NIP` (customer VAT ID). Falls back to authenticated user ID for logged-in requests.
2. **Redis-Based Counter**: Uses atomic `INCR` + `EXPIRE` (via `SET` with `NX` & `EX`) per key:  
   `rl:{tenant}:{type}:{identifier}:{window}`  
   (e.g., `rl:saas:ksef:192.168.1.1:1min`)
3. **Quota Service**: Checks limits per window (e.g., 100 req/min/IP, 10 req/sec/NIP). Returns `Retry-After` header if exceeded.

**Queue Architecture (for overflow)**:
- On rate limit exceeded, *non-critical* requests (e.g., sync reports) are enqueued into **Redis Streams**:  
  `stream:ksef:pending:{tenant}`  
  Each message includes `retry_count`, `timestamp`, and `backoff_ms`.
- A **dedicated consumer** (`@KSeFConsumer`) processes the stream at a controlled rate (e.g., 20 req/sec via `@Scheduled(fixedDelay = 50ms)` or using Kafka with rate-limited partitions).
- Failed retries use exponential backoff: `delay = base * 2^retry_count` (capped at 10 min).

**Pseudocode**:
```java
@GetMapping("/ksef/request")
public ResponseEntity<?> submitToKseF(@RequestHeader(value="X-NIP", required=false) String nip,
                                      HttpServletRequest request) {
    String clientIp = getClientIp(request);
    String key = String.format("rl:saas:ksef:%s:%s", clientIp, "1min");

    long count = redisTemplate.opsForValue().increment(key);
    if (count == 1) redisTemplate.expire(key, 60, TimeUnit.SECONDS);

    if (count > ipRateLimit) {
        // Queue for later processing (example: report sync)
        kafkaProducer.send("ksef-queue", buildMessage(clientIp, nip));
        return ResponseEntity.status(503)
            .header("Retry-After", String.valueOf(60))
            .body("Rate limit hit. Request queued.");
    }

    // Proceed to KSeF API
    return ksefClient.call(nip);
}
```

**Backoff Strategy**:
- Use `RetryTemplate` with `ExponentialBackOffPolicy`:  
  `setInitialInterval(500)`, `setMultiplier(2.0)`, `setMaxAttempts(5)`.
- On HTTP 429 from KSeF: extract `Retry-After`, wait, then retry *once*. Beyond that, fallback to queue.

**Tradeoffs**:
- ✅ *Accuracy*: Redis provides precise, distributed limits (vs. local in-memory).  
- ❌ *Latency*: Redis round-trip adds ~2–5 ms (mitigated with connection pooling & pipeline).
- ✅ *Fairness*: Per-tenant/IP+NIP prevents noisy neighbors.  
- ❌ *User Experience*: Queuing delays (seconds → minutes) may frustrate users expecting instant responses.  
- ✅ *Resilience*: Queue + backoff buffers against KSeF outages.  
- ❌ *Complexity*: Two-tier logic (immediate vs. queue) and retry state management increase code/test surface.  
- ❌ *Memory*: Large queues risk Redis memory pressure; implement TTL + dead-letter queue (DLQ) for failed messages.

**Summary**: This design prioritizes correctness and fairness over immediate response time, aligning with KSeF’s strict rate limits. Production tuning requires monitoring Redis hit rate, queue lag, and KSeF’s actual response latencies.
