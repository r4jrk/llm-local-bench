## Rate Limiting System Design for KSeF Integration

As a senior backend engineer, I recommend an asynchronous, layered approach utilizing a dedicated queue and a sophisticated rate-limiting service to ensure resilience and optimal API usage against KSeF.

### 1. Architecture Overview

The system should decouple the immediate request handling from the external API call, preventing upstream failures from cascading back into the application core.

**Components:**

1. **API Service Layer:** Accepts user requests, validates input, and publishes tasks.
2. **Message Queue (Kafka/RabbitMQ):** Acts as the buffer and sequencing mechanism.
3. **Rate Limiter Processor (Spring Consumer):** Consumes tasks from the queue, determines rate limits (IP/NIP), applies backoff, and executes the external call.
4. **Distributed Cache (Redis):** Used to maintain the current rate limit counters (sliding window or token bucket).

### 2. Implementation Details

#### A. Rate Limiting (IP + NIP)

The `RateLimiterProcessor` must check Redis for rate limits using a compound key: `RATE:{IP}:{NIP}`.

1. **Key Structure:** `RedisKey: IP_Address:NIP_Identifier`
2. **Mechanism:** Use a **Token Bucket** algorithm managed by Redis to track available requests per window (e.g., 10 requests per 60 seconds).
3. **Logic:** Before sending a request, the processor calls a Redis command (e.g., `INCR` or Lua script) to check if the limit has been breached.

#### B. Retry and Backoff Strategy

If the KSeF API returns a 429 ("Too Many Requests") or a 5xx error, the system must not simply fail.

1. **Backoff:** The processor must read the `Retry-After` header provided by the KSeF API.
2. **Queue Handling:** Instead of retrying immediately, the task is **re-queued** with a specific **delay TTL (Time-To-Live)** matching the `Retry-After` value.
3. **Jitter:** To prevent the "thundering herd" problem when multiple tasks hit the limit simultaneously, a small, randomized delay (jitter) should be added to the exponential backoff calculation.

### 3. Proposed Queue Architecture

**Message Queue:** Kafka is ideal due to its high throughput, persistent log, and ability to handle consumer groups and delayed messaging (though implementing strict delay often requires a specific Kafka mechanism or a dedicated scheduling service like Quartz).

**Flow:**
1. API Service $\rightarrow$ Kafka Topic (`ksef_requests`).
2. `RateLimiterProcessor` consumes messages.
3. **If Limit Breached/Error:** Instead of failing, the processor uses the queue's underlying mechanisms (or a separate dedicated Retry Topic) to publish the message back after a calculated delay, ensuring the consuming worker only attempts the API call when the rate limit window permits.

### 4. Pseudocode: Processing Loop

```pseudocode
FUNCTION process_ksef_request(message):
    IP = message.ip
    NIP = message.nip
    
    IF check_rate_limit(IP, NIP):
        // Token available
        execute_api_call(message)
    ELSE:
        // Limit exceeded
        delay = calculate_backoff(message, attempt_count)
        
        IF KSEF_ERROR_429_RECEIVED:
            delay = get_retry_after_header(response) 
            // Re-queue the message to a scheduled topic
            kafka_producer.send(message, topic="ksef_retries", delay_ms=delay)
        ELSE:
            // Handle non-rate-limit errors (e.g., 500)
            retry_task(message, delay=delay)
```

### 5. Tradeoffs Discussion

| Feature | Tradeoff | Discussion |
| :--- | :--- | :--- |
| **Kafka Over RabbitMQ** | Complexity vs. Throughput/Scalability | Kafka offers superior partitioning and durability for massive scale, making it better suited for enterprise SaaS use. |
| **Redis (Token Bucket)** | Speed vs. Consistency | Using Redis Lua scripting ensures atomic rate limit checks, minimizing race conditions that simple `INCR` operations might face. |
| **Asynchronous Queueing** | Latency vs. Resilience | Sacrifices immediate, synchronous API response time for vastly superior reliability and predictable throughput, which is critical when dealing with external rate limits. |
| **Delay Topic/Re-queuing** | Implementation Overhead vs. Accuracy | While adding complexity, it is mandatory. Polling/retrying locally is inefficient; re-queuing guarantees the worker only attempts the call when the resource is available. |
