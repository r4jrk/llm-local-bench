 To design a robust rate limiting system for the given SaaS application integrating with KSeF API, we will implement a solution that handles IP and NIP (Network Interface Protocol) rate limits, includes a retry/backoff strategy, and proposes an efficient queue architecture.

1. IP & NIP Rate Limiting:
   The system will maintain a list of client IP addresses and associated rate limit counters, with separate limits for each IP. Additionally, we'll consider NIP rate limiting by checking the unique MAC addresses (NIP) from within each subnet to avoid overloading individual clients behind a NAT or load balancer.

2. Retry/Backoff Strategy:
   When a client exceeds its rate limit, it will be temporarily blocked and placed in a retry queue. The system should implement an exponential backoff strategy where the time between retries is doubled for each failed attempt, up to a maximum limit. Once the retry period elapses or the client's rate limit resets, the client can resume API requests.

3. Queue Architecture:
   A message-driven approach will be used with Redis as the underlying queue system. Each client will have its own priority queue, with messages containing details about the API call (method, URL, request body, etc.) and necessary rate limit information. The application can then process messages from queues in a FIFO order or using a custom priority mechanism to prioritize requests based on factors like subscription tier, response time, or client importance.

  Map<String, Long> lastRequestTimes; // IP addresses as keys with timestamp values

  void addRequest(String clientIP, Message request) {
    if (isRateLimited(clientIP)) {
      clients.get(clientIP).add(request);
      return blockClient();
    }

    updateRateLimit(clientIP);
    processRequest(clientIP, request);
  }

  boolean isRateLimited(String clientIP) {
    Long limit = rateLimits.getOrDefault(clientIP, DEFAULT_RATE_LIMIT);
    Long currentRequests = clients.getOrDefault(clientIP, new ConcurrentLinkedQueue<>()).size();
    if (currentRequests >= limit) return true;
    lastRequestTimes.put(clientIP, System.currentTimeMillis());
    return false;
  }

    long current = rateLimits.getOrDefault(clientIP, DEFAULT_RATE_LIMIT);
    long newLimit = Math.min(current * RATE_LIMIT_MULTIPLIER, MAX_RATE_LIMIT);
    rateLimits.put(clientIP, newLimit);
  }

    return clients.get(clientIP).poll();

5. Tradeoffs:
   - The solution relies on Redis, which increases the system's complexity and cost due to its additional dependency. However, it provides fast access times for rate limit checks and message queues.
   - Implementing NIP rate limiting allows for better fairness among clients but may lead to increased computational overhead when calculating MAC addresses and maintaining subnet associations.
   - The custom priority queue might require additional development efforts compared to using off-the-shelf solutions, but it provides more flexibility in managing requests based on the specific needs of the application.
