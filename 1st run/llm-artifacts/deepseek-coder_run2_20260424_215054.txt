Designing a rate limiting system for an SaaS application with Spring Boot and the KSeF API requires careful consideration of multiple factors such as latency, throughput capacity, error rates etc., to ensure optimal performance while mitigating potential security risks like Denial-of-Service (DoS) attacks. 

The proposed rate limiting system would include IP+NIP based request counting and retry/backoff strategies for handling throttling issues within the KSeF API, using a queue architecture to manage incoming requests efficiently due to frequent bursts of high volume data or transactions while maintaining latency under normal load conditions. 

Here's an outline on how this might be implemented:

1) **IP+NIP Based Request Counting** - This strategy is used for rate limiting by identifying the IP address and NIP (Network Identifier). Both are necessary in order to identify whether a request originated from within KSeF or another API.  For each of these, we will maintain an internal counter that can be incremented whenever requests come into play based on our given criteria such as timeframe for which the limit is applied and rate limits per minute/hour etc., all in accordance with pre-agreed policies & regulations (like GDPR or CCPA).
   - Once a request comes, we will first check if this IP+NIP combination has reached its assigned quota.  If it does not reach the limit then allow requests to proceed; otherwise block further attempts until rate is reset by our system after reaching throttling condition(s) ie., retry mechanism or backoff strategy (like exponential/jittering delays).
   - The IP+NIP combination's remaining quota can be updated as and when a request comes.  After the time period expiry, this will reset to its initial limit again enabling efficient requests in next set of intervals with no delay for blocked attempts or retry mechanism (like exponential backoff strategy).
   - This approach maintains simplicity but may introduce more complexity due to handling concurrent access and ensuring thread safety if used correctly.  Also the algorithm can become complex as data size increases, reducing performance efficiency by caching frequently accessed IPs/NIP combinations for quicker lookup times in a database or similar storage mechanism is advised before using this strategy again at high volume rates (upto many requests per second).
   
2) **Retry and Backoff Strategy** - Implementing retry mechanisms to handle throttling issues within KSeF API can help reduce latency. Retrying failed attempts periodically or with a fixed delay between each attempt until the maximum number of retries are reached (which is decided by our system configuration). 
   If requests exceed rate limits, we will use backoff strategy ie., introduce delays in retry mechanism to give other calls an opportunity before finally failing. This helps prevent flooding service with too many attempts at once or if not handled properly it can lead into DOS attacks further degrading performance of the system and even causing API failure (like throttling).
   
3) **Queue Architecture** - Use a queue data structure to manage requests based on priority i.e., high volume transactions should be processed before low-volume ones, maintaining latency under normal load conditions with efficient processing time by using queues that can serve the incoming request faster than any other method of handling it at present (FIFO). 
   This also ensures fast response times for clients making requests and less delay in case if throttling occurs. Moreover this approach reduces complexity as we don't have to manage concurrency yourself, only worry about queueing up the incoming request without blocking others till rate is reset or retries are exhausted (which can be implemented using various data structures like LIFO Queue for FIFO requests etc.).

5) **Discuss Tradeoffs** - The main trade-off here would come from ensuring optimal performance under normal load conditions while maintaining security in all scenarios (IP+Nip based rate limiting with retry/back off). In terms of latency, using a Queue to manage requests can be beneficial since we might not have concurrency issues at present. However this also means our system must ensure that the queued up request will reach its target location and handle it in time frame as per policy ie., high volume transactions should precedence over low volumes (FIFO). Moreover, if retry attempts exceed maximum number of retries then we have to implement error handling on each call.
   A more advanced implementation can be designed where the queue is not only processed at once but in batches and also with priority for certain requests that are causing longer wait times than others ie., high latency transactions should precedence over low-latency ones (FIFO). This strategy could offer better throughput performance under heavy load.
   In terms of scalability, this solution can be adapted to suit large scale deployments as it would not require substantial changes in existing components and the system design itself will remain flexible for use with KSeF API by adding more features or capabilities based on requirements (like rate limiting across different services).  Also a distributed caching layer like Redis could enhance performance especially when handling high volume data.
   Finally, while designing this solution it is crucial to consider all the security implications too such as protecting against DOS attacks and ensuring only authorized requests are processed in terms of authorization checks before processing (if any) using JWT tokens or similar authentication methods for SaaS applications can be utilized here.  All these considerations ensure a robust design that meets every possible scenario while being secure, efficient under normal load conditions with good performance at peak times and easy to scale up further in future when required by the business needs accordingly (like increasing rate limits).
