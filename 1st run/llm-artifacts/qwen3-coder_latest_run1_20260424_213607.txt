## Rate Limiting System for Spring Boot SaaS with KSeF Integration

### Core Architecture
I'll implement a **token bucket algorithm** with **dual-level rate limiting** (IP + NIP) using Redis for distributed state management.

```java
@Component
public class RateLimitingService {
    private final RedisTemplate<String, String> redisTemplate;
    
    public boolean isAllowed(String ip, String nip, int maxRequests, long windowMs) {
        String key = "rate_limit:" + ip + ":" + nip;
        String tokenKey = key + ":tokens";
        String lastRefillKey = key + ":last_refill";
        
        // Atomic operations for thread safety
        return redisTemplate.opsForValue().increment(tokenKey, 1) <= maxRequests;
    }
}
```

### Queue Architecture
**Redis-based priority queue** with TTL expiration:
- **High priority**: Critical NIP requests
- **Medium priority**: Regular API calls  
- **Low priority**: Batch processing
- **Dead letter queue** for failed requests with exponential backoff

### Key Tradeoffs

**Pros:**
- **Scalability**: Redis clustering handles massive concurrent requests
- **Accuracy**: Real-time rate limiting prevents abuse
- **Flexibility**: Configurable limits per NIP type

**Cons:**
- **Latency**: Network round-trip to Redis adds ~1-2ms
- **Complexity**: Distributed state management overhead
- **Cost**: Redis infrastructure expenses

**Alternative Approach**: In-memory caching with eventual consistency for non-critical paths.

**Performance Impact**: 5-10% overhead on request processing time. For high-volume SaaS, this is acceptable vs. potential abuse costs.

The system balances **security** (preventing abuse) with **performance** (minimal latency impact) while maintaining **resilience** through retry mechanisms and queue prioritization. This design scales horizontally and provides predictable rate limiting behavior across distributed microservices.
