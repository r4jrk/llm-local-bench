To design a resilient rate-limiting system for KSeF (the Polish National Repository of Structured Electronic Documents), we must move beyond simple request dropping. Since KSeF integrations are typically asynchronous (e.g., sending invoices), we need a **Buffered Token Bucket** architecture.

### Architectural Design

I propose a distributed architecture using **Redis** for state management and **RabbitMQ** for request buffering and retries.

1.  **Identification Layer:** We use a composite key: `ratelimit:{IP}:{NIP}`. This allows us to enforce strict limits per tenant (NIP) while also preventing IP-based DoS attacks.
2.  **The Interceptor (Token Bucket):** Using Redis, we implement a **Sliding Window Log** or **Token Bucket**. Before any KSeF call, the service checks the current count for the `IP+NIP` key.
3.  **The Queueing Strategy (The Buffer):** 
    *   **Primary Queue:** High-priority queue for immediate processing.
    *   **Retry/Delay Queue:** If a limit is hit or a `429 Too Many Requests` is received from KSeF, the task is moved to a **RabbitMQ Delayed Message Exchange**.
4.  **Retry/Backoff Logic:** We implement **Exponential Backoff with Jitter**. This prevents "thundering herd" problems when the KSeF window resets.

### Pseudocode

```java
@Service
public class KSefIntegrationService {

    @Autowired private RedisTemplate<String, Integer> redisTemplate;
    @Autowired private RabbitTemplate rabbitTemplate;

    public void processInvoice(InvoiceRequest request) {
        String key = String.format("rate_limit:%s:%s", request.getIp(), request.getNip());
        
        if (isRateLimited(key)) {
            // Push to delayed queue to retry later
            scheduleRetry(request, 30); // Initial 30s delay
            return;
        }

        try {
            executeKSefCall(request);
        } catch (KSefRateLimitException e) {
            // Handle 429 from KSeF: Exponential backoff
            int retryCount = request.getRetryCount() + 1;
            long delay = calculateBackoff(retryCount);
            scheduleRetry(request.incrementRetry(), delay);
        }
    }

    private boolean isRateLimited(String key) {
        long currentCount = redisTemplate.opsForValue().increment(key);
        if (currentCount == 1) redisTemplate.expire(key, Duration.ofMinutes(1));
        return currentCount > MAX_ALLOWED_PER_MINUTE;
    }

    private void scheduleRetry(InvoiceRequest request, long delaySeconds) {
        rabbitTemplate.convertAndSend("ksef.retry.exchange", "retry.key", request, msg -> {
            msg.getMessageProperties().setDelay((int) delaySeconds * 1000);
            return msg;
        });
    }
}
```

### Tradeoffs

*   **Consistency vs. Latency (Redis):** Using Redis provides a single source of truth for all Spring Boot instances (essential for SaaS), but adds a network hop (latency) to every request. I chose Redis because "local" limits would allow an attacker to bypass limits by hitting different pod IPs.
*   **Reliability vs. Complexity (Queueing):** Introducing RabbitMQ ensures we never "lose" an invoice when limits are hit, but it introduces complex failure modes (e.g., what happens if the queue grows too large?). We must implement **Dead Letter Queues (DLQ)** to alert engineers when retries fail permanently.
*   **Cost vs. Precision:** The Sliding Window algorithm is more precise than Fixed Window (avoiding bursts at window boundaries) but requires more memory in Redis to store timestamps. Given the critical nature of KSeF compliance, the memory overhead is a justified trade-off.
