**Rate Limiting System Design: **  
The primary aspect to consider for a Rate-Limited SaaS API is the handling of IP and NIP combinations, ie., User Identity or Namespace Identifiers (NIPS). The system should be designed in such way that it can handle multiple requests from different sources simultaneously.  It's beneficial if you have to support thousands/millions users concurrently using your API over a network of IP addresses and NIP identities, therefore considering the high load scenario is necessary for this design:
- Handle User ID / Namespace Identifiers Rate Limiting by implementing Leaky Bucket Algorithm. 
     * A "Leak" in terms means not releasing all resources immediately due to concurrent request rates higher than a certain limit and it's set based on the user identity or namespace identifier (NIP).  
    - In this case, each NIP would have an allocation point for requests which defines how many calls can be made within 'tick time'.  For instance: if there are two users with identities A & B who call your API at different rates. User "A" will start getting rate limited after the first tick and block all further activity until it finishes its current request on NIP, then user "B".
- Include a retry/backoff strategy which handles cases where requests fail due to temporary errors (rate limiting exceeded).  This would mean that at some point in time users should be allowed back into the system after completing their previous calls. Retry logic could use exponential or linear backoffs according with error type and might also implement a queue for retries like Kafka based on message queues to deal out retry attempts asynchronously, thus avoiding heavy load over HTTP requests while they are being handled by other components in the system (efficiency).
- Propose Queue Architecture using Apache's Kafka. It should be configured such that when a rate limit is exceeded for an NIP/ID combination it can store this request and send them to queue after certain time, which could then either run again or wait in the system until available resources are freed up before processing further requests from same user who requested earlier once more due.
