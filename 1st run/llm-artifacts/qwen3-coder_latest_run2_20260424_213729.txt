## Rate Limiting System for Spring Boot SaaS with KSeF API

### Core Architecture
I'll implement a **token bucket algorithm** with **dual-level rate limiting** (IP + NIP) using Redis for distributed state management.

```java
@Component
public class RateLimitingService {
    private final RedisTemplate<String, String> redisTemplate;
    
    public boolean isAllowed(String ip, String nip, int maxRequests, long windowSeconds) {
        String key = "rate_limit:" + ip + ":" + nip;
        String bucketKey = key + ":bucket";
        
        // Atomic operation to check and update
        String script = "local current = redis.call('GET', KEYS[1]) " +
                       "if current == false then " +
                       "  redis.call('SET', KEYS[1], 1) " +
                       "  redis.call('EXPIRE', KEYS[1], ARGV[1]) " +
                       "  return 1 " +
                       "else " +
                       "  local count = tonumber(current) " +
                       "  if count >= tonumber(ARGV[2]) then return 0 else " +
                       "    redis.call('INCR', KEYS[1]) " +
                       "    return 1 " +
                       "  end " +
                       "end";
        
        return redisTemplate.execute(
            (RedisCallback<Boolean>) connection -> 
                connection.eval(
                    script.getBytes(),
                    ReturnType.BOOLEAN,
                    1,
                    bucketKey.getBytes(),
                    String.valueOf(windowSeconds).getBytes(),
                    String.valueOf(maxRequests).getBytes()
                )
        );
    }
}
```

### Queue Architecture
Implement a **priority queue system** with **retry/backoff** using **Redis Streams** and **Spring Retry**:

```java
@Service
public class KSeFQueueService {
    @Retryable(
        value = {Exception.class},
        maxAttempts = 3,
        backoff = @Backoff(delay = 1000, multiplier = 2)
    )
    public void processKSeFRequest(String requestId, String payload) {
        if (!rateLimitingService.isAllowed(getClientIp(), getClientNip(), 100, 60)) {
            throw new RateLimitExceededException("Rate limit exceeded");
        }
        
        // Queue processing with Redis Streams
        redisTemplate.opsForStream()
            .add("ksef_requests", 
                 Map.of("requestId", requestId, "payload", payload));
    }
}
```

### Key Tradeoffs

**Memory vs. Accuracy**: Redis-based solution offers better accuracy than in-memory but requires network latency. For SaaS, this is acceptable.

**Complexity vs. Performance**: Dual rate limiting adds complexity but provides better security. Consider using **leaky bucket** for consistent throughput vs. **token bucket** for burst handling.

**Retry Strategy**: Exponential backoff prevents overwhelming KSeF API while maintaining system responsiveness. **Dead letter queues** should handle persistent failures.

**Scalability**: Redis clustering handles horizontal scaling. **Circuit breaker** pattern protects against cascading failures.

**Monitoring**: Track rate limit violations, retry counts, and processing times through metrics. **Alerting** should trigger when 80% of limits are reached.

This design balances system reliability with performance, ensuring KSeF API compliance while maintaining SaaS scalability.
