## Rate Limiting System Design for KSeF Integration

As a senior backend engineer, I recommend an asynchronous, queuing-based architecture to ensure resilience and compliance when integrating with the KSeF API. Direct, synchronous calls from a Spring Boot service are too fragile and would lead to cascading failures under rate limit pressure.

### 1. Architecture Overview (The Queue Layer)

The core of the system will be a message queue (e.g., **RabbitMQ** or **Kafka**).

**Flow:**
1. **Ingress:** The primary Spring Boot endpoint receives a request.
2. **Queueing:** Instead of calling KSeF directly, the service immediately serializes the request payload and publishes it to a designated message queue (`ksef_requests`).
3. **Processing:** A dedicated, isolated worker service (the **RateLimiter Worker**) consumes messages from this queue.
4. **Rate Limiting:** The worker manages access to KSeF by tracking usage limits using a high-speed, distributed cache (like **Redis**).

### 2. Handling Dual Rate Limits (IP + NIP)

The Redis key structure is critical for tracking consumption:

*   **Key Structure:** `rate_limit:{type}:{identifier}:{window_seconds}`
*   **IP Limit Tracking:** `rate_limit:ip:CLIENT_IP:60` (Tracks requests by source IP).
*   **NIP Limit Tracking:** `rate_limit:nip:NIP_IDENTIFIER:60` (Tracks requests by the specific NIP provided).

The worker service checks both keys before executing any external API call.

### 3. Retry and Backoff Strategy

When the KSeF API returns a `429 Too Many Requests` status (or a custom rate limit error code), the worker must not fail; it must adapt.

1. **Detection:** The worker catches the 429 status.
2. **Wait Time Calculation:** The response headers (e.g., `X-RateLimit-Retry-After`) are checked first. If missing, a **Full Jitter Backoff** algorithm is used (e.g., $T = \text{random}(\text{min\_wait}, \text{max\_wait})$).
3. **Requeueing:** The message is not dropped. Instead, it is **republished to the queue with a time delay**. This is best handled by a dedicated **Delay Queue** mechanism in the message broker, or by having a sophisticated scheduling service.

### 4. Pseudocode for Worker Logic

```pseudocode
FUNCTION process_message(message):
    payload = deserialize(message)
    ip = get_source_ip()
    nip = payload.nip_identifier

    // Check Limits (Atomic Redis operation)
    IF redis.check_rate_limit(ip, 'IP') OR redis.check_rate_limit(nip, 'NIP'):
        log("Rate limit exceeded. Backing off.")
        retry_delay = calculate_backoff(message.headers) 
        republish_to_delay_queue(message, retry_delay)
        RETURN

    // Execution
    try:
        response = KsefClient.call(payload)
        process_success(response)
    except RateLimitExceededError:
        retry_delay = calculate_backoff(response.headers)
        republish_to_delay_queue(message, retry_delay)
    except Exception as e:
        log_failure(message, e)
        # Handle terminal failures (e.g., invalid payload)
```

### 5. Discussion of Tradeoffs

| Feature | Benefit | Tradeoff |
| :--- | :--- | :--- |
| **Asynchronous Queue (Kafka/RabbitMQ)** | Decouples services, manages backpressure, enables guaranteed delivery and eventual processing. | Adds latency; the end user receives an immediate "queued" response, not the final result. |
| **Redis Rate Limiting** | Extremely fast, atomic counter checks; supports distributed state management across multiple workers. | Requires maintaining a dedicated, highly available Redis cluster. |
| **Delay/Dead Letter Queues** | Ensures messages are eventually retried and allows for monitoring of permanent failures (DLQ). | Adds complexity to the queue topology and requires specialized consumer logic. |

**Conclusion:** This architecture maximizes resilience and throughput, treating the external API as a constrained resource managed by a robust, self-healing queue system.
