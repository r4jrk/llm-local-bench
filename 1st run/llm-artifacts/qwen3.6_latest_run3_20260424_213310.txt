A Spring Boot SaaS integrating with KSeF must strictly honor per-IP and per-NIP rate limits while maintaining multi-tenant isolation. The system decouples request generation from transmission using a durable message queue, enforced by a distributed Redis-based rate limiter and exponential backoff.

**Rate Limiting (IP+NIP)**
KSeF enforces separate counters per IP and per NIP. We implement a sliding-window counter in Redis using a Lua script for atomic reads/writes. Each tenant’s requests are scoped to `(tenant_id, ip, nip)` to prevent cross-tenant leakage. A `RateLimiterService` returns `true` only when both the IP and NIP buckets have sufficient tokens. If either is exhausted, requests are deferred rather than dropped.

**Queue Architecture**
Producers publish `KsefSubmissionJob` to RabbitMQ with headers: `ip`, `nip`, `tenant_id`, `priority`. A pool of consumers routes to tenant-scoped internal queues to guarantee isolation. Workers pull jobs, verify Redis buckets, and forward to KSeF. Overloaded or rejected jobs are requeued with a delayed delivery policy (RabbitMQ DLX) rather than blocking the main thread.

**Retry & Backoff**
On HTTP 429 or transient failures, the worker applies exponential backoff with jitter: `delay = min(2^retry * 1s * random(0.8, 1.2), 60s)`. Retry count is persisted in the message payload. After 5 attempts, the job routes to a DLQ for audit/manual retry. Successful submissions acknowledge immediately and clear associated Redis counters.

**Pseudocode**
```pseudocode
// Redis Lua: atomic sliding-window check
LUA = "local key = KEYS[1]; local now = redis.call('TIME')[1]; " +
      "local window = redis.call('ZREMRANGEBYSCORE', key, 0, now-60); " +
      "return redis.call('ZCARD', key) < tonumber(ARGV[1])";

function tryAcquire(tenant, ip, nip):
    ipKey = "ksel:ip:" + tenant + ":" + ip
    nipKey = "ksel:nip:" + tenant + ":" + nip
    return redis.eval(LUA, [ipKey], 10) and redis.eval(LUA, [nipKey], 50)

function processJob(job):
    retries = 0
    while retries < MAX_RETRIES and not tryAcquire(job.tenant, job.ip, job.nip):
        queue.defer(job, 2000 * retries) // simplified
    
    resp = http.post("/ksef/submit", job.payload)
    if resp.status == 429:
        queue.defer(job, exponentialBackoff(retries))
```

**Tradeoffs**
- *Redis vs Relational DB:* Redis provides atomic O(1) operations and low latency but requires persistence tuning and adds cluster management overhead.
- *Sliding Window vs Token Bucket:* Sliding window aligns better with KSeF’s exact request counters but increases Redis memory usage per tenant. Token bucket allows controlled bursts at the cost of drift.
- *Queue Routing:* Tenant-scoped queues prevent noisy-neighbor degradation but increase operational complexity. A single queue with priority headers reduces overhead but risks starvation.
- *Backoff Jitter:* Essential to avoid thundering herds on KSeF’s infra, though it increases tail latency and queue depth.
- *SaaS Isolation:* Hard tenant scoping is non-negotiable. Shared rate limits or weak key derivation could cause a high-volume client to block a low-volume tenant, violating SLAs.
