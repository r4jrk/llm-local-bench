To ensure KSeF compliance and prevent KSeF-imposed IP/NIP bans, we must implement a **Distributed Token Bucket** architecture. Since KSeF limits are often tied to specific tax identifiers (NIP) and originating IPs, a centralized state is mandatory.

### 1. Architecture Overview
We will use a **Producer-Consumer pattern** with a **Redis-backed Rate Limiter** and **RabbitMQ** for buffering.

1.  **Ingress Layer (Spring Boot Interceptor):** Every request is intercepted. We generate a composite key: `rl:{ip}:{nip}`.
2.  **Rate Limiter (Redis + Lua):** We use a Lua script in Redis to atomically check and decrement tokens. This prevents race conditions across multiple service instances.
3.  **Buffer Layer (RabbitMQ):** If a request exceeds the limit, instead of returning `429 Too Many Requests` to the user, we return `202 Accepted` and push the task into a **Delay Queue**.
4.  **Worker Layer:** A dedicated consumer pulls tasks. If it encounters a 429 from KSeF, it implements **Exponential Backoff with Jitter**.

### 2. Pseudocode

**Rate Limiter Interceptor (The Guard)**
```java
public class KSeFRateLimitInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest req, ...) {
        String key = "rl:" + req.getRemoteAddr() + ":" + req.getHeader("X-NIP");
        
        if (redisTemplate.execute(luaScript, List.of(key), limit, window)) {
            return true; // Proceed to Controller
        } else {
            // Move to background processing
            messageQueue.sendToRetryQueue(new KSeFTask(req.getBody()), calculateDelay());
            throw new RateLimitExceededException("Request queued for later processing");
        }
    }
}
```

**Worker Service (The Executor)**
```java
@RabbitListener(queues = "ksef.retry.queue")
public void processKSeFRequest(KSeFTask task) {
    try {
        ksefClient.sendInvoice(task.getPayload());
    } catch (KSeFResponse429Exception e) {
        long nextRetry = calculateBackoff(task.getRetryCount());
        // Re-queue with a delay (Dead Letter Exchange pattern)
        rabbitTemplate.convertAndSend("ksef.delayed.exchange", task.getRetryCount() + 1, 
            task, message -> {
                message.getMessageProperties().setDelay((int) nextRetry);
                return message;
            });
    }
}
```

### 3. Tradeoffs

| Feature | Approach | Pro | Con |
| :--- | :--- | :--- | :--- |
| **Consistency** | Redis Lua | Atomic, prevents over-limit bursts. | Adds dependency on Redis; network latency. |
| **User Experience** | Async (202 Accepted) | Users aren't "blocked"; high throughput. | **Eventual Consistency**: User doesn't know *exactly* when it's done. |
| **Reliability** | RabbitMQ Delay Queues | Resilience against KSeF downtime. | Increased infrastructure complexity and cost. |

### 4. Senior Engineer's Summary
To scale this, the "Retry Queue" must use **Dead Letter Exchanges (DLX)** with TTL. This allows us to "park" messages for 1, 5, or 15 minutes without consuming CPU. The biggest risk is **Queue Congestion**: if KSeF goes down for hours, the queue grows. We must implement a **Circuit Breaker** (Resilience4j) that stops the Producer from even hitting the Interceptor if the KSeF error rate exceeds a threshold, protecting our system from memory exhaustion.
