For a Spring Boot SaaS integrating with KSeF, I recommend a decoupled, distributed architecture centered around Redis for rate limiting and Kafka for message queuing. This ensures scalability, fault tolerance, and compliance with strict API constraints.

Rate limits must be enforced per tenant (NIP) and per client IP to prevent cross-tenant leakage and comply with KSeF’s quotas. We’ll use a Sliding Window Counter via Spring Data Redis with compound keys: `ratelimit:{ip}:{nip}`. A Redis Lua script ensures atomic check-and-increment operations. Incoming REST controllers delegate to an async service that validates the request and publishes to Kafka, returning a 202 Accepted immediately. NIPs are normalized and hashed in a GDPR-compliant tenant mapping table; lookups occur post-queue to avoid exposing raw PII in Redis keys.

For retry logic, we’ll integrate Resilience4j with WebClient. On HTTP 429 or transient network failures, Resilience4j applies exponential backoff: `baseDelay=1s, maxDelay=30s, multiplier=2.0, jitter=0.3`. Backoff is applied at the consumer layer, not the producer, preventing retry storms. Jitter is critical to avoid thundering herds when multiple tenants resume after a KSeF outage.

The queue architecture uses Kafka topic `ksef-requests` with partitioning keyed by tenant ID. A dedicated consumer group processes messages in a single-threaded, ordered fashion per partition. The consumer atomically checks the Redis rate limit, executes the KSeF payload via WebClient with circuit breaking, and publishes results to `ksef-responses`. Unrecoverable payloads route to a Kafka DLQ for manual inspection or scheduled reprocessing.

```java
// Producer
kafkaTemplate.send("ksef-requests", tenantKey, new Request(ip, nip, payload));

// Consumer
@KafkaListener(topics = "ksef-requests")
public void process(ConsumerRecord<String, Request> record) {
    String key = "rl:" + record.value().ip() + ":" + record.value().nip();
    Long remaining = redis.opsForValue().increment(key);
    if (remaining > RATE_LIMIT) {
        backoff.apply();
        return; // Kafka retains until retry or DLQ
    }
    var response = ksefClient.post(record.value().payload());
    kafkaTemplate.send("ksef-responses", response);
}
```

**Tradeoffs:** 
Redis vs PostgreSQL: Redis provides O(1) latency and atomic Lua scripts but requires careful memory management and cluster operations. PostgreSQL with advisory locks reduces infra cost but introduces contention and serialization bottlenecks under high throughput. 
Queue durability vs complexity: Kafka guarantees at-least-once delivery and elastic backpressure but adds operational overhead. RabbitMQ simplifies deployment but lacks native topic-based partitioning, risking message drift across tenants.
IP+NIP enforcement: Compound keys prevent quota abuse but multiply Redis memory footprint. We mitigate this via keyspace TTL (`windowSize + 1s`) and consistent hashing to Redis Cluster. NIP normalization is mandatory to avoid duplicate keys from formatting variations.
Consumer parallelism: Single-threaded per partition guarantees strict per-tenant request ordering but caps throughput. We scale partitions dynamically during onboarding and use consumer scaling with sticky partition assignment to maintain order while maximizing parallelism. This architecture balances tenant isolation, infra efficiency, and graceful degradation under KSeF’s strict limits.
