As a senior backend engineer, I recommend designing a **Producer-Consumer architecture** utilizing a message queue (like Kafka) to decouple the core SaaS logic from the external KSeF API rate limitations. This ensures that the SaaS remains responsive even if the external API is throttled.

### System Design

**1. Rate Limiting (IP + NIP)**
We must enforce two distinct limits:
*   **IP Limit:** Track usage per calling IP address.
*   **NIP Limit:** Track usage per specific customer NIP (National Identification Number).

We will use **Redis** for centralized, high-speed rate limit tracking. The keys stored in Redis would incorporate both the source identifier and the limit group (e.g., `rate:NIP:{NIP}:count` and `rate:IP:{IP}:count`).

**2. Queue Architecture (Kafka)**
The SaaS service (Producer) does not call KSeF directly. Instead, it publishes request messages to a dedicated Kafka topic (e.g., `ksef_requests`).

*   **Producer:** Publishes the request payload (NIP, endpoint, data).
*   **Consumer (Worker Service):** A dedicated, internal worker service consumes messages from the topic. This worker service handles all external API calls, rate limiting checks, and retries, preventing the main business logic from being affected by external throttling.

**3. Retry/Backoff Strategy**
The Consumer Service implements the retry logic:

*   **Immediate Check:** Before calling KSeF, the Consumer checks Redis. If limits are exceeded, the message is *not* processed immediately.
*   **Backoff:** If a `429 Too Many Requests` is received, the consumer must not fail the message. Instead, it updates the message with a calculated delay (exponential backoff: $2^n$ seconds) and either places it back on a **Delayed Queue** (if using a system that supports it) or fails it back to a **Dead Letter Queue (DLQ)** with metadata that schedules reprocessing after the calculated interval.

### Pseudocode (Consumer Service Logic)

```pseudo
function consume_message(message):
    retry_count = message.headers.get('retry_count')
    delay_seconds = message.headers.get('delay')

    if delay_seconds and delay_seconds > 0:
        schedule_reprocess(message, delay_seconds)
        return

    try:
        if check_limits(message.NIP, message.IP):
            response = call_ksef_api(message)
            publish_result(response)
        else:
            // Rate limit hit (Redis check failed)
            raise RateLimitExceeded()
    except RateLimitExceeded as e:
        new_delay = calculate_exponential_backoff(retry_count)
        update_message_delay(message, new_delay)
        send_to_delayed_queue(message)
    except Exception as e:
        // Handle permanent failures (e.g., bad data)
        send_to_dlq(message, e)
```

### Tradeoffs Discussion

| Feature | Advantage | Disadvantage |
| :--- | :--- | :--- |
| **Queue Architecture (Kafka)** | High resilience, graceful handling of bursts, full decoupling. | Increased complexity, eventual consistency (results are not instant). |
| **Redis State Management** | Extremely low latency for limit checks, reliable atomic increments. | Requires robust infrastructure maintenance (Redis cluster). |
| **Exponential Backoff** | Prevents immediate hammering of the API, respects external rate limits. | Can significantly delay processing, requiring robust monitoring of the "Delayed Queue." |

This architecture prioritizes **resilience and throughput** over immediate consistency, which is critical when integrating with external, rate-limited third-party APIs like KSeF.
