To handle KSeF’s strict constraints, we must decouple the request submission from the actual API call. A synchronous approach will lead to thread exhaustion and high error rates.

### 1. Architecture Overview
I propose a **Distributed Asynchronous Worker** pattern using **Redis** for rate tracking and **RabbitMQ/Kafka** for orchestration.

**Components:**
- **API Gateway/Service:** Validates requests and pushes them to a "Pending" queue.
- **Rate Limiting Layer:** A Redis-backed sliding window counter.
- **Worker Pool:** Consumes the queue, checks limits, and executes calls.
- **Retry Exchange:** Handles failed requests with delayed retries.

### 2. Rate Limiting Strategy
Since KSeF limits both by **IP** and **NIP**, we must track two separate keys in Redis per request:
- `limit:ip:{ip_address}`
- `limit:nip:{nip_number}`

I recommend using a **Lua script** in Redis to ensure atomicity. The script checks if both counters are below the threshold before incrementing and allowing the request.

### 3. Queue & Retry Architecture
1. **Main Queue:** Stores the payload and metadata (NIP, IP).
2. **Worker Logic:** 
   - Peek/Fetch message.
   - Check Redis limits for both IP and NIP.
   - If limited $\rightarrow$ Re-queue with a `delay` header or move to a "Wait" queue.
   - If allowed $\rightarrow$ Call KSeF.
3. **Retry/Backoff:** Upon receiving a `429 Too Many Requests` or `5xx` error, the message is moved to a **Retry Exchange** with **Exponential Backoff + Jitter** (e.g., $2^n + \text{random\_ms}$). After $X$ attempts, move to a Dead Letter Queue (DLQ) for manual intervention.

// RateLimiterService.java
public boolean isAllowed(String nip, String ip) {
    // Redis Lua script executes: 
    // IF (get(nip) < MAX_NIP && get(ip) < MAX_IP) { incr(nip); incr(ip); return true; }
    return redisTemplate.execute(luaScript, List.of(nip, ip));
}

// KSeFWorker.java
@RabbitListener(queues = "ksef_queue")
public void process(KSefRequest request) {
    if (!rateLimiter.isAllowed(request.getNip(), request.getIp())) {
        throw new RateLimitExceededException("Re-queueing..."); 
    }

    try {
        kSeFClient.sendInvoice(request);
    } catch (HttpClientErrorException.TooManyRequests e) {
        // Trigger exponential backoff
        retryService.scheduleRetry(request, attempt + 1);
    }
}

### 5. Trade-offs

| Trade-off | Decision | Reasoning |
| :--- | :--- | :--- |
| **Latency vs. Throughput** | Throughput | By using a queue, we increase end-to-end latency for a single invoice but maximize total system throughput and stability. |
| **Redis vs. Local Cache** | Redis | Local caching (Caffeine) doesn't work in a multi-node SaaS; a centralized store is required to prevent aggregate limit breaches. |
| **Strict vs. Soft Limits** | Soft | We implement soft limits in Redis slightly *below* KSeF's hard limits to account for network jitter and avoid triggering KSeF's aggressive blocking. |
| **Ordering** | Eventual | Rate limiting via re-queuing may result in invoices being sent out of sequence. If strict ordering is required, we must implement one queue per NIP (highly complex) or use a Sequencer. |
